Zigbee Relay Status

Group: Zigbee. Type: PnZigbeeRelayStatus.

Receive-only decoder for a single Zigbee on/off endpoint exposed by a Zigbee2MQTT bridge. Where the Zigbee Switch fuses inbound state, the on-canvas slider and the outbound set builder into one widget, this node is the factored read-half: it has no latch and no slider, it just listens for the device's reported state and forwards it as a value message. Pair it with a Zigbee Relay Command when you want independent read and write branches.

Behaviour

Zigbee2MQTT publishes a device's full state as a JSON object on the bare friendly-name topic, e.g. on zigbee2mqtt/livingroom_lamp: 

{ "state": "ON", "linkquality": 142, "brightness": 254 }

The node filters to exactly zigbee2mqtt/<friendly-name> (no /set or /availability suffix), reads the top-level state member, and — only for the literal "ON" / "OFF" values — reshapes the message into the canonical pipnode contract:

MemberValue
data.value1.0 (ON) or 0.0 (OFF)
data.successtrue
data.devicethe configured friendly name
data.output<name>: state ON

The original envelope and data.payload (linkquality, brightness, …) are left intact for nodes that want the detail. A "TOGGLE" value, or a state object with no top-level state member, is dropped silently so a noise burst cannot synthesise a false transition. Unlike the Zigbee Switch's inbound path, this node does emit — it is a pure observer with no latch to sync, and it never rewrites the topic to a set command, so there is no feedback-loop risk.

Configuration
PropertyDescription
friendly-name Zigbee2MQTT friendly name (e.g. livingroom_lamp) the filter listens for, matched as the exact tail of zigbee2mqtt/<friendly-name>. Mandatory — an empty value puts the node in the red “configuration required” state and rejects every message, since there is no sensible default device to pick on the user's behalf.
Wiring example
MQTT Source (url = tcp://mqtt.homelab.local:1883, topic = zigbee2mqtt/#)
    → Zigbee Relay Status (friendly-name = livingroom_lamp)
        → LED / Debug / Graph

Whenever Z2M reports the lamp's state — whether the change came from the Z2M frontend, an automation, or a Zigbee Relay Command elsewhere in the worksheet — this node forwards a value message a downstream indicator recognises.

Z2M base topic. The default Z2M base_topic is zigbee2mqtt, which this node pins. If your Z2M instance uses a non-default base_topic, adjust Z2M back to the default or wire the explicit, factored chain (generic filter + payload decoder) until a configurable-prefix variant lands.
