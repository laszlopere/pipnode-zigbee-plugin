Zigbee Water Leak

Group: Zigbee. Type: PnZigbeeWaterLeak.

Edge filter and reshaping node for Zigbee water-leak sensors bridged through Zigbee2MQTT. A leak sensor publishes its state on zigbee2mqtt/<friendly_name> with a data.payload JSON object carrying a boolean water_leak member, alongside the usual battery and radio attributes: 

{ "water_leak": true,  "battery": 93, "linkquality": 233, "tamper": false, "voltage": 2800 }
{ "water_leak": false, "battery": 93, "linkquality": 229, "tamper": false, "voltage": 2800 }

Drop this node between a Zigbee Source (or a plain MQTT Source) and any downstream node to turn the raw stream of publishes into a clean event on each real leak transition: 

Zigbee Source → Zigbee Water Leak → Notify / Debug / Switch / …
Why an edge filter

The sensor republishes its whole payload on every routine update — a battery report, a link-quality refresh — and almost all of those carry water_leak: false. Passing through the bare value would therefore fire constantly. What actually matters is the transition: the moment a leak begins (false → true) or clears (true → false). This node remembers the last water_leak value it saw per device and only forwards a message when that value changes; the repeated same-state updates in between are dropped.

What it does
StepBehaviour
Filter Only messages whose data.payload is an object carrying a boolean water_leak are considered; everything else (other device types, bridge traffic, bare-string availability) is dropped silently. Of those, only a message that changes the remembered state for that device, and whose edge the mode selects, is forwarded.
Reshape Survivors are stamped into the canonical message contract so a downstream LED, Debug, Graph or Notify reads the same shape it gets from any other node:
  • data.success = true
  • data.value = 1.0 on a leak begin, 0.0 on a leak end
  • data.device = the resolved sensor name
  • data.output = a human-readable line, e.g. Water leak detected on Kitchen / Water leak cleared on Kitchen
The original envelope and data.payload (the battery, linkquality, tamper, voltage, and any injected device block) are left untouched, so a downstream node can still inspect the detail.
First message after the node starts

A device's first publish only seeds the remembered state — nothing is forwarded for it. With no prior value there is no transition to report, and seeding quietly avoids firing a spurious “begin” just because the node was dropped (or the worksheet reloaded) while a leak happened to be active. The next genuine change then fires normally.

How the sensor is named

The name in data.output / data.device — also the key used to track state per device — is resolved in this order:

  1. data.payload.device.friendlyName — the identity block a Zigbee Source injects by default (inject-device-info), the human-friendly Z2M name.
  2. The friendly-name tail of the envelope topic (zigbee2mqtt/<tail> → <tail>); for an unrenamed device this is its IEEE address, e.g. 0xa4c13860befb28ec.
  3. The bare topic, then unknown, as last resorts.
Configuration
PropertyDescription
friendly-name Optional sensor filter. When set, only edges from this sensor pass; matched against the injected data.payload.device.friendlyName (when an upstream Zigbee Source injects device info) or, failing that, the friendly-name tail of the envelope topic (zigbee2mqtt/<tail> — the bare device id for an unrenamed device), so you can enter either form. Empty (the default) tracks every water-leak sensor.
mode Which transitions to forward:
  • Leak begins — only the false → true edge.
  • Leak ends — only the true → false edge.
  • Both (the default) — either edge.
Wiring example
MQTT/Zigbee Source (topic = zigbee2mqtt/#)
    → Zigbee Water Leak (mode = Leak begins)
        → Notify (title = "{{output}}")

When water bridges the sensor it publishes zigbee2mqtt/0xa4c13860befb28ec with {"water_leak":true,…}; the Zigbee Source forwards it (and, by default, injects the device block); the Zigbee Water Leak node sees the false → true transition, sets value=1.0 / success=true and an output line like Water leak detected on Kitchen, and the Notify node pops it on screen. The subsequent water_leak:false battery reports are dropped until the leak genuinely clears.
