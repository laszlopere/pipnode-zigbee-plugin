Zigbee Switch

Group: Zigbee. Type: PnZigbeeSwitch.

Bidirectional device node for a single Zigbee on/off endpoint exposed by a Zigbee2MQTT bridge. Merges the on-canvas slide switch (Switch) with the inbound state filter and the outbound set command builder into one node so the canonical “show the device state, let the user flip it” worksheet wires up as three nodes: 

MQTT Source → Zigbee Switch (friendly-name = livingroom_lamp) → MQTT Sink

Subclass of Switch, so the worksheet's slider-click routing, hit-testing and painting all come from the same implementation — only the encode / decode shape and the device-side topic contract are Zigbee-flavoured.

Direction split
DirectionBehaviour
Inbound (MQTT → latch) A message whose envelope topic is exactly zigbee2mqtt/<friendly-name> (the canonical Z2M state topic, no /set or /availability suffix) and whose data.payload JSON object carries "state": "ON" or "OFF" syncs the latch silently — the slider position updates but nothing is emitted downstream. The inbound is an authoritative state observation, not a transit message that should fan out; forwarding it would either leak raw MQTT bytes into observer nodes or close a loop back through the sink.
User click (latch → MQTT) Clicking the slider flips the latch and emits a fresh Z2M command message: envelope topic zigbee2mqtt/<friendly-name>/set, data.payload a JSON object { "state": "ON" } or { "state": "OFF" }, plus the standard data.value / data.success / data.device / data.output mirror so a downstream MQTT Sink publishes the literal Z2M expects, and a parallel observer (LED, Debug, Graph) sees a value form it recognises.

Startup command is opt-in. The base Switch announces its latch state once when a worksheet loads, so downstream nodes learn its position without a click. Because the Zigbee Switch's output is a physical device command, this is off by default: the device's true state instead arrives over the inbound zigbee2mqtt/… path, which syncs the slider silently, so opening a worksheet never actuates hardware. Set enforce-on-startup if you want the worksheet to be the source of truth and force the device to the saved position on load.

Configuration
PropertyDescription
friendly-name Zigbee2MQTT friendly name (e.g. livingroom_lamp) the node binds to. Inbound: matched as the exact tail of zigbee2mqtt/<friendly-name>. Outbound: stamped into zigbee2mqtt/<friendly-name>/set. Mandatory by design — an empty value puts the node in the red “configuration required” state, rejects every inbound message, and refuses to emit on user-click, since silently defaulting would risk reading or flipping the wrong physical device.
on Inherited from Switch. Current latch position (persisted in the worksheet so the slider parks back the way the user left it on reload).
enforce-on-startup When enabled, the node emits a zigbee2mqtt/<friendly-name>/set command for the saved latch position once shortly after the worksheet loads, forcing the physical device to match the worksheet on startup. Off by default — opening a worksheet never actuates the device; the latch tracks the device's reported state instead. Enable it when the worksheet (not the device) is the source of truth for the desired state.
Wiring example
MQTT Source (url = tcp://mqtt.homelab.local:1883, topic = zigbee2mqtt/#)
    → Zigbee Switch (friendly-name = livingroom_lamp)
        → MQTT Sink (url = tcp://mqtt.homelab.local:1883)

Click the slider on the canvas; the node emits zigbee2mqtt/livingroom_lamp/set with payload {"state":"ON"} or {"state":"OFF"}; MQTT Sink publishes; Z2M relays the command over the Zigbee mesh; the bulb responds; Z2M publishes the new state on zigbee2mqtt/livingroom_lamp; the MQTT Source forwards it; Zigbee Switch's inbound path sees its own state confirmed and silently moves on (no echo storm). Flipping the bulb externally (the Z2M frontend, an automation, another flow) follows the same return path and the slider tracks the new state.

Z2M base topic. The default Z2M base_topic is zigbee2mqtt, which this node pins. If your Z2M instance is configured with a non-default base_topic, adjust Z2M back to the default or wire the explicit, factored chain (generic filter + Switch + raw MQTT payload encoder) until a configurable-prefix variant lands.
