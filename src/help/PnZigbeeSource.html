Zigbee Source

Group: Zigbee. Type: PnZigbeeSource.

A Zigbee2MQTT-flavoured MQTT Source. Subclass of MQTT Source (PnMqtt); the broker connect / subscribe / reconnect / visual-state machinery comes from the base unchanged. Two specialisations on top:

  • The inherited subscribe-topic defaults to zigbee2mqtt/#, so a freshly-dropped node already points at the Z2M topic tree.
  • An early topic filter (PnMqttClass::accept_topic override) optionally drops zigbee2mqtt/bridge/logging on the network thread before a PnMessage is built — cheap noise rejection that skips the JSON parse / message build / main-thread marshal cost.
  • A late message hook (PnMqttClass::process_message override) caches the device inventory Z2M publishes on zigbee2mqtt/bridge/devices and grafts a device object onto every per-device state publish's data.payload — the same shape Z2M emits when configured with include_device_information: true, but produced client-side at no Z2M / network cost.
Why filter logging?

Zigbee2MQTT publishes its own log lines to zigbee2mqtt/bridge/logging as {"level":"info|warning|error","message":"..."}. That channel is what the Z2M web UI's Logs tab consumes; useful there, pure noise in a worksheet that only cares about device traffic and lifecycle events. Filtering it out at the source keeps downstream nodes' Debug panes and graphs readable without changing the subscribe filter.

The other bridge/... topics (event, state, devices, groups, definitions, response/...) carry structured data — device join / leave events, online / offline, retained inventories, RPC responses — and always pass through. Users who want a stricter cut can narrow the subscribe-filter directly (e.g. zigbee2mqtt/+, which excludes the whole bridge/ subtree).

Device-info injection

Z2M publishes its full device inventory on zigbee2mqtt/bridge/devices with the retained flag, so a fresh subscriber receives it automatically on connect and again whenever the device set changes. The node parses each such publish into an in-memory cache keyed by friendly_name. On every subsequent per-device state publish (zigbee2mqtt/<friendly_name>) the cached record is spliced into data.payload under the key device.

Two independent opt-ins each contribute their own non-overlapping set of fields to the device object. Each can be on or off on its own — turn off everything you don't need so the important things stay easy to find:

infocaps What lands under data.payload.device
offoff nothing — no device member is added at all.
onoff Z2M-compatible identity block only.
offon Capability / category / description block only — the publish's envelope topic already names the device, so omitting the identity scalars is fine.
onon Union of both, identity fields first.
Identity block — inject-device-info (default on)

Matches Z2M's include_device_information output verbatim, so a downstream consumer written against that Z2M shape works unchanged: 

{
  "state": "ON",
  "power": 0,
  "voltage": 239,
  ...
  "device": {
    "friendlyName":     "0xa4c138a4c9626802",
    "ieeeAddr":         "0xa4c138a4c9626802",
    "networkAddress":   34662,
    "type":             "Router",
    "powerSource":      "Mains (single phase)",
    "manufacturerName": "_TZ3000_okaz9tjs",
    "modelID":          "TS011F",
    "model":            "TS011F_plug_1",
    "softwareBuildID":  "1.0.5"
  }
}

Fields that are not in bridge/devices (applicationVersion, stackVersion, zclVersion, hardwareVersion, manufacturerID — Z2M sources these from its in-process zigbee-herdsman device object) are omitted.

Capability block — inject-device-capabilities (default off)

A separate field set lifted from Z2M's bridge/devices definition sub-object, so a downstream reader can answer “what is this, what can it do, what can I configure” from one place. When enabled with identity off, the spliced device object contains only these keys (the publish's envelope topic already identifies the device): 

"device": {
  "category":     "plug",
  "description":  "Smart plug (with power monitoring)",
  "vendor":       "TuYa",
  "supportsOta":  false,
  "exposes":      [ ...full capability tree, verbatim... ],
  "options":      [ ...Z2M device-config options, verbatim... ]
}
  • category — best-effort one-word label synthesised from the exposes tree and power source. Possible values: plug (mains-powered switch), switch, light, lock, cover, fan, climate, remote (exposes a top-level action), sensor (all exposes are read-only), or the fallback device.
  • description / vendor / supportsOta — Z2M's human-readable definition fields lifted up to the device block.
  • exposes — the structured capability tree verbatim from definition.exposes: per-feature access bitmap (1=published, 2=settable, 4=gettable), value ranges, enum value lists, units, labels. This is what answers “what commands can I send” and “what does this device report”.
  • options — the Z2M-side device-configuration knobs verbatim from definition.options.

Default off because the exposes and options trees can run several KB per device, making each state publish proportionally bigger. Enable when a downstream node actually needs to introspect device commands / configuration. Both field sets are precomputed (as disjoint cache objects) when bridge/devices arrives, so toggling either property at runtime takes effect on the next per-device publish without rebuilding the cache.

Drop the node, point it at the broker, run. The Debug pane fills with device state publishes (zigbee2mqtt/<friendly_name>) and bridge lifecycle events (zigbee2mqtt/bridge/event, .../state, ...) but stays free of the per-line log chatter from zigbee2mqtt/bridge/logging.

Relationship to MQTT Source

Drop a plain MQTT Source instead when you need to subscribe outside the Z2M tree, or when you want explicit control over every topic the node forwards. Use Zigbee Source for the common case of "give me Zigbee traffic, hide the log spam".
