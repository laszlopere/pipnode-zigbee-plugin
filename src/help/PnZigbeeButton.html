Zigbee Button

Group: Zigbee. Type: PnZigbeeButton.

Filter and reshaping node for a single Zigbee wireless button — e.g. a SONOFF SNZB-01P — bridged through Zigbee2MQTT. It is the button-flavoured sibling of the Zigbee Remote node: structurally almost identical, it exists so a single-button device gets its own palette entry, naming and output wording. Such a button publishes each press on zigbee2mqtt/<friendly_name> with a data.payload JSON object whose action member names the gesture: 

{ "action": "single", "battery": 100, "linkquality": 185 }

The SNZB-01P emits single, double and long. Drop this node between a Zigbee Source (or a plain MQTT Source) and any downstream node to turn those raw publishes into clean, canonical pipnode messages: 

Zigbee Source → Zigbee Button → Debug / Notify / Switch / …
What it does
StepBehaviour
Filter Only messages whose data.payload is an object carrying a non-empty string action pass — i.e. an actual press. The idle / state-only publishes Z2M interleaves, and the empty-action “released” frames some converters emit, are dropped (nothing is forwarded).
Reshape Survivors are stamped into the canonical message contract so a downstream LED, Debug, Graph or Notify reads the same shape it gets from any other node:
  • data.success = true
  • data.value = 1.0
  • data.output = a human-readable line, e.g. Button kitchen_button pressed (action: single)
The original envelope and data.payload (the action, battery, linkquality, and any injected device block) are left untouched, so a downstream node can still inspect the detail.
How the button is named

The name in data.output is resolved in this order:

  1. data.payload.device.friendlyName — the identity block a Zigbee Source injects by default (inject-device-info), which is the human-friendly Z2M name.
  2. The friendly-name tail of the envelope topic (zigbee2mqtt/<tail> → <tail>); for an unrenamed device this is its IEEE address, e.g. 0xf044d3fffef9779d.
  3. The bare topic, then unknown, as last resorts.

For the friendliest output, keep this node downstream of a Zigbee Source with device-info injection enabled (its default).

Configuration

Both properties are optional filters. Left empty (the default) they are wildcards, so the node forwards every press. Set either to narrow the node to one specific button and/or one specific gesture — they are ANDed.

PropertyDescription
friendly-name When set, only presses from this button pass. Matched against the injected data.payload.device.friendlyName (when an upstream Zigbee Source injects device info) or, failing that, the friendly-name tail of the envelope topic (zigbee2mqtt/<tail> — the bare device id for an unrenamed device), so you can enter either form. Empty matches every button.
action When set, only presses whose data.payload.action equals this exact value pass (e.g. single, double, long). Empty matches every action.
Wiring example
MQTT/Zigbee Source (topic = zigbee2mqtt/#)
    → Zigbee Button (action = "single")
        → Notify (title = "{{output}}")

A single press publishes zigbee2mqtt/0xf044d3fffef9779d with {"action":"single",…}; the Zigbee Source forwards it (and, by default, injects the device block); the Zigbee Button recognises the press, sets value=1.0 / success=true and an output line like Button kitchen_button pressed (action: single), and the Notify node pops it on screen.
