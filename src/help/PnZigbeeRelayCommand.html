Zigbee Relay Command

Group: Zigbee. Type: PnZigbeeRelayCommand.

Send-only command builder for a single Zigbee on/off endpoint exposed by a Zigbee2MQTT bridge. Where the Zigbee Switch fuses inbound state, the on-canvas slider and the outbound set builder into one widget, this node is the factored write-half: it has no latch and no slider, it just turns an incoming value message into a device command. Pair it with a Zigbee Relay Status when you want independent read and write branches.

Behaviour

The node consumes the canonical pipnode value message (any node that emits a numeric data.value — an Inject, a Switch, a threshold filter, …), treats value > 0.5 as ON and the rest as OFF, and rewrites the envelope so a downstream MQTT Sink publishes straight to the device's command topic:

MemberValue
topiczigbee2mqtt/<friendly-name>/set
data.payloadJSON object { "state": "ON" } or { "state": "OFF" }
data.value1.0 (ON) or 0.0 (OFF)
data.successtrue
data.devicethe configured friendly name
data.output<name>: command state ON

The payload is set as a structured JSON object (not a pre-stringified string): MQTT Sink serialises it back to JSON bytes on the wire, so a Debug node in between sees the structured form rather than opaque bytes. This is exactly the shape the Zigbee Switch emits on a click, so the factored write-half is wire-compatible with the fused node. A message that carries no usable numeric data.value leaves the device alone.

Configuration
PropertyDescription
friendly-name Zigbee2MQTT friendly name (e.g. livingroom_lamp) to command, stamped into zigbee2mqtt/<friendly-name>/set. Mandatory by design — an empty value puts the node in the red “configuration required” state and rejects every message, since silently defaulting would risk flipping the wrong physical device.
Wiring example
Inject (value = 1) → Zigbee Relay Command (friendly-name = livingroom_lamp)
    → MQTT Sink (url = tcp://mqtt.homelab.local:1883)

The Inject fires a value; this node emits zigbee2mqtt/livingroom_lamp/set with payload {"state":"ON"}; MQTT Sink publishes; Z2M relays the command over the Zigbee mesh; the bulb responds. Wire a Zigbee Relay Status off the same MQTT Source to observe the confirmed state independently.

Z2M base topic. The default Z2M base_topic is zigbee2mqtt, which this node pins. If your Z2M instance uses a non-default base_topic, adjust Z2M back to the default or wire the explicit, factored chain (payload encoder + Rewrite) until a configurable-prefix variant lands.
