ZNP Ping

Group: Zigbee. Type: PnZnpPing.

Source-only diagnostic node. Holds a TI CC2652 / CC1352 Zigbee coordinator dongle (e.g. AVATTO GW70-MQTT) open over USB serial and, at a configurable interval, sends a Z-Stack SYS_PING request over the TI UNPI framing. On the first successful reply it also issues SYS_VERSION so the worksheet sees what firmware is on the stick. Anything else the coordinator emits in the meantime — boot banners, ZDO indications once the network is up, unsolicited AF_INCOMING_MSG events — is forwarded verbatim as a raw “frame” message. Enough to confirm “is this dongle alive and what is it saying” without standing up a Zigbee2MQTT daemon.

Properties
device
Serial-port path of the coordinator. Defaults to /dev/ttyUSB0. Prefer the stable /dev/serial/by-id/usb-Silicon_Labs_CP2102_USB_to_UART_Bridge_Controller_* symlink for setups with more than one USB-serial device. The port is opened at 115200 8N1, no flow control, with flock(LOCK_EX) and TIOCEXCL so a second instance pointed at the same path fails with EBUSY.
interval
Seconds between SYS_PING attempts (default 5). Set to 0 to disable polling — the node then only forwards unsolicited frames.
Output messages

Three distinct shapes, distinguished by the topic envelope field:

topic = "znp/ping"
One per successful SYS_PING round-trip. Fields: success (TRUE), capabilities (16-bit capability bitmap from the coordinator), output (a human-readable summary).
topic = "znp/version"
Emitted after the first successful ping per device-open. Fields: transport, product, major, minor, maintenance, revision, and a human-readable output string of the form Z-Stack X.Y.Z (transport T, product P, build R).
topic = "znp/frame"
One per unsolicited / unexpected ZNP frame received. Fields: cmd0, cmd1, subsystem (cmd0 & 0x1F), type (cmd0 & 0xE0 — 0x40 is AREQ, 0x60 SRSP, 0x20 SREQ), length, and the data_hex / output hex dump of the payload bytes.

On any error (open failure, EBUSY, write failure, port HUP, ping reply timeout) the node emits a success = FALSE message carrying output with the human-readable reason, switches the icon to the red warning glyph, and tears the port down. The next property change or worksheet save will trigger a fresh open attempt.

Limitations

This node owns the serial port exclusively for the lifetime of the node instance, the same way the Meshtastic node does — only one ZNP Ping per dongle, and you cannot run a parallel Zigbee2MQTT daemon against the same stick while this node holds it. The proper fix is host-managed shared profile sessions (see item 28 in the pipnode TODO), which would let multiple Zigbee nodes share one queued connection. Until that lands, treat this as a debug-only single-owner tool.
